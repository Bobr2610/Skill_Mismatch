# 3. Принцип Питера: некомпетентность как следствие продвижения

## 3.1. Формулировка принципа

В 1969 году канадский педагог **Лоуренс Дж. Питер** сформулировал одно из самых цитируемых наблюдений в теории менеджмента:

> *«В иерархии каждый служащий стремится достичь уровня своей некомпетентности.»*

Суть принципа: в иерархической организации сотрудники получают повышение на основе результатов на **текущей** должности, а не готовности к **следующей**. Лучший программист становится тимлидом, лучший тимлид — менеджером проекта, и так далее — до тех пор, пока человек не оказывается на позиции, где его компетенций уже недостаточно. На этом уровне он «застревает», поскольку:

- Его не повышают дальше (нет результатов)
- Его редко понижают (организационная инерция, трудовое законодательство, репутационные издержки)

## 3.2. Эмпирические подтверждения

Принцип Питера долгое время воспринимался как сатира, однако ряд современных исследований подтвердил его эмпирически:

### Исследование Benson, Li, Shue (2019)

Анализ данных 53 000+ сотрудников отдела продаж в 214 фирмах показал:

- Сотрудники, **наиболее эффективные как продавцы**, после повышения до менеджеров показывали **худшие результаты** в управлении командой
- Каждый рост продаж на удвоенную величину (pre-promotion) был связан с **падением производительности подчинённых** на 7,5% после назначения такого менеджера

### Исследование Galeotti & Merlino

В модели с асимметричной информацией авторы показали, что конкурентный рынок труда **систематически** порождает эффект Принципа Питера:

- **Средние по способностям** работники оказываются «повышены» в более престижные фирмы и получают непропорционально высокие ренты
- **Более способные** работники остаются в менее продуктивных фирмах, недооценёнными
- Этот результат возникает **не из-за ошибок**, а как равновесный исход рыночных механизмов

## 3.3. Механизм действия в ИТ-организациях

В ИТ-индустрии Принцип Питера проявляется через характерные паттерны:

### Паттерн 1: «Лучший кодер — худший менеджер»

| Этап | Junior Dev | Mid Dev | Senior Dev | Tech Lead | Eng. Manager |
|---|---|---|---|---|---|
| **Результат** | OK | OK | OK | Риск | Провал |
| **Фокус** | Код | Код | Код + дизайн | Код + люди | Только люди |

Навыки, делавшие человека отличным разработчиком (глубокий фокус, перфекционизм, работа в одиночку), прямо **противоречат** требованиям менеджерской позиции (делегирование, коммуникация, управление неопределённостью).

### Паттерн 2: «Раздувание грейда»

Компании создают «индивидуальные карьерные треки» (IC track), чтобы обойти Принцип Питера, но на практике:

- Титулы растут быстрее компетенций (title inflation)
- «Staff Engineer» в стартапе может соответствовать «Mid Developer» в крупной компании
- Нет объективных метрик, подтверждающих соответствие грейду

### Паттерн 3: «Невидимый герой»

Сотрудник, который:

- Поддерживает критическую legacy-систему
- Знает все edge-case'ы продукта
- Проводит большинство code review

...часто остаётся на уровне Middle/Senior, потому что его работа **невидима** для традиционных систем оценки. Его вклад не отражается ни в количестве новых фич, ни в «громких» проектах.

## 3.4. Следствия Принципа Питера

| Следствие | Описание |
|---|---|
| **Потеря компетенций** | Лучшие исполнители уходят из зоны своей максимальной эффективности |
| **Демотивация команды** | Некомпетентное руководство снижает мотивацию подчинённых |
| **Каскадный эффект** | Каждое неудачное повышение порождает вакансию, заполняемую по тому же ошибочному принципу |
| **Организационный склероз** | Критические позиции заняты людьми, не способными их эффективно выполнять |
| **Потеря талантов** | Недооценённые компетентные сотрудники уходят в другие компании |

## 3.5. Связь с Skill Mismatch

Принцип Питера является **внутриорганизационным генератором** Skill Mismatch:

$$\text{Skill Mismatch (internal)} = \text{Peter Principle} + \text{Subjective Evaluation} + \text{Org. Inertia}$$

Если **внешний** Skill Mismatch возникает при найме (неверное соответствие «кандидат — позиция»), то **внутренний** — при продвижении (неверное соответствие «текущие навыки — требования новой роли»). Оба механизма усиливают друг друга, создавая самоподдерживающийся цикл неэффективности.
