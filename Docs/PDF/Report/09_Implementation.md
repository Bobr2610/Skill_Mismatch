# 9. Реализация интеллектуальной системы оценки компетенций

## 9.1. Концепция решения

### 9.1.1. Системный анализ проблемы

Традиционные методики имеют межэкспертную надёжность r = 0.45–0.52 (Salgado, 2019); идиосинкратический эффект (Foster, 2024) объясняет 50–72% вариации. 72% сотрудников не доверяют аттестации (Deloitte, 2025).

### 9.1.2. Теоретическая основа

Решение переходит от субъективной оценки к **объективному анализу рабочих артефактов** в ИТ:

- Артефакты контроля версий: коммиты, pull requests, code review
- Артефакты управления задачами: cycle time, lead time, throughput
- Артефакты качества кода: метрики SonarQube, cognitive complexity
- Артефакты верификации: результаты тестирования, coverage metrics

## 9.3. Инструментальный комплекс верификации

### 9.3.1. AI-генерация тестов и верификация

Ключевой элемент — AI-генерация тестов и верификация кода.

**Метрики верификации:**

| Метрика | Описание | Вес |
|---------|----------|-----|
| Pass Rate | Отношение пройденных тестов к общему числу | 0.40 |
| Coverage | Покрытие кода тестами | 0.25 |
| Edge Case Coverage | Покрытие граничных случаев | 0.20 |
| Error Handling | Корректность обработки исключений | 0.15 |

**Сравнительный анализ подходов:**

| Параметр | Традиционный code review | AI-генерация тестов |
|----------|-------------------------|-------------------|
| Объективность | Субъективное мнение | Фактический результат |
| Полнота | Зависит от внимательности | Систематическое покрытие |
| Воспроизводимость | Вариативность | Идентичность тестов |
| Масштабируемость | Линейная | Константное время |

### 9.3.2. Анализ артефактов контроля версий

**Git-коммиты:** анализируются количественные характеристики (объём, частота, temporal distribution), качественные характеристики (типология изменений: feature, bugfix, refactor), структурные характеристики (размер изменений, cohesion metrics).

**Pull Requests:** измеряются временные параметры (время создания, время ревью), глубина анализа (количество комментариев), итеративность (число переоткрытий).

**Code Review:** оценивается активность участия, конструктивность обратной связи, оперативность реагирования, способность выявления критических дефектов.

### 9.3.3. Интеграция с SonarQube

| Метрика | Описание | Порог |
|---------|----------|-------|
| Cognitive Complexity | Сложность понимания кода | < 15 |
| Maintainability Index | Индекс поддерживаемости | > 80 |
| Code Duplication | Процент дублирования | < 3% |
| Test Coverage | Покрытие тестами | > 80% |
| Technical Debt Ratio | Технический долг | < 5% |

### 9.3.4. DORA-метрики

Система интегрирует стандартизированные метрики производительности DevOps:

**Deployment Frequency**: Elite (ежедневно), High (еженедельно), Medium (ежемесячно), Low (реже)

**Lead Time for Changes**: Elite (< 1 часа), High (1 день — 1 неделя), Medium (1 неделя — 1 месяц), Low (> 1 месяца)

**Change Failure Rate**: Elite (0–5%), High (6–15%), Medium (16–30%), Low (> 30%)

**Mean Time to Recovery**: Elite (< 1 часа), High (< 1 дня), Medium (1 день — 1 неделя), Low (> 1 недели)

### 9.3.5. LLM-генерация тестов

Использование современных языковых моделей обеспечивает высокий уровень автоматизации:

| Функция | Точностные характеристики |
|---------|--------------------------|
| Генерация тест-кейсов | 65–70% (HumanEval) |
| Идентификация дефектов | 75–80% (SWE-bench) |
| Code Review | 85–90% (CodeReviewBench) |
| LLM-as-a-Judge | 82–85% |

## 9.4. Математический аппарат

### 9.4.1. Рекуррентная модель обновления

Фундаментальное уравнение системы — формула рекуррентного пересчёта с учётом временного затухания компетенций:

$$P_{new} = \min(100, \max(0, P_{current} \times K + M))$$

Где:
- $P_{new}$ — обновлённое значение параметра (0–100)
- $P_{current}$ — текущее значение
- $K$ — коэффициент затухания (0.4–0.75)
- $M$ — вклад за отчётный период

### 9.4.2. Позиционные коэффициенты затухания

| Позиция | K | Период полувыведения (мес.) |
|---------|---|----------------------------|
| Junior | 0.40 | 0.76 |
| Middle | 0.50 | 1.00 |
| Senior | 0.60 | 1.36 |
| Lead | 0.70 | 1.94 |
| Director | 0.75 | 2.41 |

### 9.4.3. Формулы параметров профиля

**Implementation Quality:**
$$IQ = PR \times 0.40 + COV \times 0.25 + EC \times 0.20 + EH \times 0.15$$

**Task Effectiveness:**
$$TE = \frac{\sum passed_i}{\sum total_i} \times 100$$

**Традиционные параметры:**
- $SV = \frac{\sum \lambda_i \cdot Usage_i}{RequiredStack}$ — Stack Versatility
- $\Delta MI = MI_{post} - MI_{pre}$ — Maintainability Delta
- $CI = \frac{\sum Comments}{LeadTime_{avg}}$ — Collaboration Impact
- $RF = \frac{Commits_{architectural} + Commits_{refactoring}}{TotalCommits}$ — Role Fidelity
- $CA = \frac{CyclomaticComplexity_{avg}}{CycleTime}$ — Cognitive Agility
- $W = \frac{E_{completed}}{N_{planned}}$ — Productivity Index

## 9.5. Алгоритмическая реализация

### 9.5.1. Пятиэтапный процесс оценки

**Этап 1: Агрегация данных**
Установление соединений с API источников, извлечение истории коммитов, PR, метрик задач за период, проверка целостности.

**Этап 2: Генерация тестов**
NLP-парсинг требований задач, LLM-генерация тест-кейсов (базовые, граничные, негативные, performance), валидация синтаксиса.

**Этап 3: Верификация решений**
Запуск тестов против кода, сбор результатов (pass/fail/error), расчёт производных метрик (PassRate, Coverage, EdgeCoverage).

**Этап 4: Расчёт параметров**
Вычисление IQ, TE, SV, CI, RF, W, CA, ΔMI, применение рекуррентной формулы обновления.

**Этап 5: Детерминация грейда**
Сопоставление профиля с эталонными матрицами, расчёт сходства, определение рекомендации (Promote/Keep/Demote), проверка истории на три последовательных снижения.

## 9.6. Гибридная модель оценки

### 9.6.1. Распределение ролей

| Функция | ИИ-система | Человек-эксперт |
|---------|-----------|----------------|
| Агрегация данных | Автоматизация | Контроль целостности |
| Анализ артефактов | Обработка | Валидация методологии |
| Расчёт метрик | Вычисление | Интерпретация |
| Детерминация грейда | Паттерн-матчинг | Контекстуальная корректировка |
| Финальное решение | Рекомендации | Утверждение |
| Контекстуальный учёт | — | Личные обстоятельства |
| Обратная связь | Отчёт | Эмоциональная поддержка |

### 9.6.2. Процедура гибридной оценки

Гибридная процедура включает шесть этапов:

1. **Инициация и сбор данных** — автоматическая агрегация из источников
2. **AI-анализ и расчёт** — обработка артефактов, вычисление параметров
3. **Мониторинг и сигнализация** — отслеживание метрик, early warning
4. **Экспертная валидация** — контекстуальная корректировка
5. **Принятие решения** — финальное утверждение кадрового решения
6. **Предоставление обратной связи** — структурированный отчёт для сотрудника

## 9.8. Ограничения и риски

| Категория | Ограничение | Митигация |
|-----------|-------------|-----------|
| Алгоритмическая предвзятость | Наследование bias из данных | Регулярный аудит |

## 9.9. Заключение раздела

Представленная интеллектуальная система оценки компетенций представляет собой комплексное инженерное решение, преодолевающее фундаментальные ограничения традиционных методик. Методологический переход от экспертного суждения к объективному анализу рабочих артефактов обеспечивает:

- **Воспроизводимость**: идентичные входные данные порождают идентичные оценки
- **Масштабируемость**: линейная сложность от числа сотрудников
- **Прозрачность**: полная трассируемость от артефактов к профилю
- **Легитимность**: объективная верификация исключает когнитивные искажения

Гибридная архитектура (ИИ + человек) синтезирует операционные преимущества автоматизации (43–58% экономии времени, устранение 33% искажений) с незаменимой ролью человеческой экспертизы.
