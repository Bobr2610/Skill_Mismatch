# 9. Решение проблемы: интеллектуальный агент оценки

## 9.1. Концепция решения

Традиционные методы оценки персонала демонстрируют системную неэффективность: межэкспертная надёжность r = 0.45–0.52 (Salgado, 2019), 72% сотрудников не доверяют системе (Deloitte, 2025), идиосинкратический эффект оценщика объясняет 50–72% вариации оценок (Foster, 2024).

Ключевая идея решения: замена субъективной оценки человека на **объективный анализ рабочих артефактов** — материальных свидетельств трудовой деятельности. В ИТ-сфере это коммиты, PR, code review, метрики качества кода.

## 9.2. Архитектура ИИ-системы оценки

### Компоненты системы

1. **Data Collector** — сбор данных из GitHub, Jira, SonarQube, таск-трекеров
2. **Artifact Analyzer** — анализ кода с помощью LLM (GPT-4, Claude)
3. **Metrics Engine** — расчёт 6 параметров профиля компетенций
4. **Grading Engine** — определение грейда по матрице критериев
5. **Decay Module** — рекуррентный пересчёт с коэффициентом затухания
6. **Recommender** — выдача рекомендаций (Promote/Keep/Demote)

### Источники данных

| Источник | Данные | Параметр |
|----------|--------|----------|
| GitHub | коммиты, PR, review | RF, CI, ΔMI |
| Jira | cycle time, типы задач | CA, W |
| SonarQube | quality gates | ΔMI |
| Slack/Teams | активность | CI |

## 9.3. Инструменты проверки эффективности

Интеллектуальный агент оценивает эффективность сотрудника через анализ множества источников данных и метрик:

### 9.3.1. Анализ Git-коммитов

Агент извлекает историю коммитов за анализируемый период и оценивает:

- **Количество коммитов** — общая активность разработки
- **Размер коммитов** — средний размер изменений (строк/файлов)
- **Частота коммитов** — распределение активности по времени (утро/день/вечер, будни/выходные)
- **Сообщения коммитов** — качество описания изменений (Conventional Commits)
- **Тип изменений** — feature, bugfix, refactor, docs, test

### 9.3.2. Анализ Pull Request

Агент анализирует каждый PR:

- **Время открытия** — скорость создания PR после начала работы
- **Время ревью** — скорость получения review
- **Количество комментариев** — Depth of Review (DoR)
- **Количество итераций** — сколько раз PR был переоткрыт
- **Размер PR** — количество изменений
- **Конфликты при merge** — частота конфликтов

### 9.3.3. Анализ Code Review

Агент оценивает качество ревью:

- **Комментарии к чужим PR** — участие в ревью коллег
- **Тональность комментариев** — конструктивная обратная связь
- **Скорость ревью** — время от создания до первого комментария
- **Глубина ревью** — замечание багов, уязвимостей, архитектурных проблем

### 9.3.4. Метрики качества кода (SonarQube)

Агент интегрируется с SonarQube для оценки:

| Метрика | Описание | Порог |
|---------|----------|-------|
| Cognitive Complexity | Сложность понимания кода | < 15 |
| Maintainability Index | Индекс поддерживаемости | > 80 |
| Duplication | Дублирование кода | < 3% |
| Coverage | Покрытие тестами | > 80% |
| Issues | количество критических/мажорных | 0 |
| Tech Debt | Технический долг | < 5% |

### 9.3.5. Метрики задач (Jira/LinearB)

Агент собирает данные из таск-трекеров:

| Метрика | Формула | Целевое значение |
|--------|---------|------------------|
| Cycle Time | Delivery - Start | < 5 дней |
| Lead Time | Deployment - Creation | < 1 дня |
| Throughput | Completed / Period | Стабильный рост |
| WIP Limit | Работа в очереди | < 5 задач |
| Blocked Time | Время в блоке | < 10% |

### 9.3.6. Анализ коммуникации (Slack/Teams)

Агент анализирует активность в мессенджерах:

- **Количество сообщений** — общая вовлечённость
- **Каналы** — участие в технических каналах
- **Время ответа** — оперативность коммуникации
- **Инициативность** — создание обсуждений vs ответы

### 9.3.7. DORA Metrics

Агент рассчитывает стандартные метрики DevOps:

1. **Deployment Frequency** — как часто деплоится код
   - Elite: ежедневно или несколько раз в день
   - High: еженедельно
   - Medium: ежемесячно
   - Low: реже чем раз в месяц

2. **Lead Time for Changes** — время от коммита до деплоя
   - Elite: < 1 часа
   - High: 1 день — 1 неделя
   - Medium: 1 неделя — 1 месяц
   - Low: > 1 месяца

3. **Change Failure Rate** — процент деплоев с ошибками
   - Elite: 0–5%
   - High: 6–15%
   - Medium: 16–30%
   - Low: > 30%

4. **Mean Time to Recovery** — время восстановления после сбоя
   - Elite: < 1 дня
   - High: < 1 дня
   - Medium: 1 день — 1 неделя
   - Low: > 1 недели

### 9.3.8. LLM-анализ кода

Агент использует LLM (GPT-4, Claude) для глубокого анализа:

- **Семантический анализ** — понимание логики кода
- **Поиск багов** — уязвимости, race conditions
- **Архитектурный анализ** — соблюдение паттернов
- **Security scan** — уязвимости безопасности
- **Performance hints** — узкие места

| Инструмент | Функция | Точность |
|-----------|---------|----------|
| GPT-4 | Анализ кода | 67% (HumanEval pass@1) |
| Claude Sonnet 4.5 | Bug fixing | 77,2% (SWE-bench Verified) |
| Claude Sonnet 4.5 | Code Review | 87,1% (CodeReviewBench) |
| GPT-4 Turbo + CoT | LLM-as-a-Judge | 84,4% |

## 9.4. Математический аппарат

### Формула рекуррентного пересчёта

$$P_{new} = \min(100, \max(0, P_{current} \times K + M))$$

Где:
- $P_{new}$ — новое значение параметра (0–100)
- $P_{current}$ — текущее значение
- $K$ — коэффициент затухания (0.4–0.75 по позиции)
- $M$ — вклад за период

### Коэффициенты затухания

| Позиция | K | t₁/₂ (месяцев) |
|---------|---|-----------------|
| Junior | 0.40 | 0.76 |
| Middle | 0.50 | 1.00 |
| Senior | 0.60 | 1.36 |
| Lead | 0.70 | 1.94 |
| Director | 0.75 | 2.41 |

### Формулы 6 параметров

1. **Stack Versatility**: $SV = \frac{\sum \lambda_i \times Usage_i}{Required\_Stack}$

2. **Maintainability Delta**: $\Delta MI = MI_{after} - MI_{before}$

3. **Collaboration Impact**: $CI = \frac{\sum Comments_{PR}}{Lead\ Time_{Review}}$

4. **Role Fidelity**: $RF = \frac{Commits_{Arch} + Commits_{Refac}}{Total\ Commits}$

5. **Cognitive Agility**: $CA = \frac{Cycle\ Time}{Cyclomatic\ Complexity}$

6. **Productivity**: $W = \frac{E}{N}$

## 9.5. Алгоритм оценки

### Этап 1: Сбор данных
```
1. Подключиться к API GitHub, Jira, SonarQube
2. Извлечь историю коммитов за период (30 дней)
3. Получить метрики PR: comments, review time, merge time
4. Собрать данные о задачах: cycle time, complexity
```

### Этап 2: Анализ артефактов
```
1. Для каждого коммита: анализ diff с помощью LLM
2. Классифицировать по типу: feature, bug, refactor, docs
3. Оценить качество кода (SonarQube metrics)
4. Рассчитать ΔMI для каждого изменения
```

### Этап 3: Расчёт параметров
```
1. Рассчитать SV на основе технологического стека
2. Рассчитать CI на основе активности в PR
3. Рассчитать RF как долю архитектурных коммитов
4. Рассчитать W как отношение completed/total hours
5. Рассчитать CA как cycle time / complexity
```

### Этап 4: Определение грейда
```
1. Сравнить профиль с эталонной матрицей
2. Определить ближайший грейд
3. Рассчитать confidence score
4. Проверить на соответствие текущему грейду
```

### Этап 5: Рекомендация
```
1. Если Params > Threshold + Delta → Promote
2. Params > Threshold → Keep
3. Params < Threshold - Delta → Demote
4. Три последовательных снижения → trigger review
```

## 9.6. Гибридная модель

### Распределение ролей

| Функция | ИИ | Человек |
|--------|----|---------|
| Сбор данных | Да | |
| Анализ артефактов | Да | |
| Расчёт метрик | Да | |
| Определение грейда | Да | |
| Финальное решение | | Да |
| Контекстуальный учёт | | Да |
| Обратная связь | | Да |

### Процесс гибридной оценки

1. **ИИ-анализ**: автоматический сбор и обработка данных, расчёт 6 параметров, формирование профиля
2. **ИИ-мониторинг**: непрерывное отслеживание метрик, сигнализация об отклонениях, раннее предупреждение
3. **Человеческая экспертиза**: финальное решение с учётом контекста, качественных факторов, личных обстоятельств
4. **ИИ-обоснование**: предоставление детализированных данных для оспаривания

## 9.8. Ограничения

- **Алгоритмическая предвзятость**: может наследовать bias из данных
- **Объяснимость**: «чёрный ящик» — сложно объяснить решение
- **Контекст**: не учитывает личные обстоятельства
- **Восприятие**: сотрудники могут не доверять ИИ

## 9.9. Заключение раздела

Интеллектуальный агент оценки — инструмент преодоления системных проблем традиционной оценки. Переход от субъективного мнения к объективным данным повышает легитимность системы.

Гибридная модель (ИИ + человек) сочетает:
- **Скорость и объективность ИИ** (43–58% экономии времени, устранение 33% искажений)
- **Контекстуальное понимание человека** (качественные факторы, эмоциональная поддержка)

Ключевые формулы ($P_{new} = P_{current} \times K + M$ с коэффициентами затухания 0.4–0.75) и алгоритм из 5 этапов обеспечивают воспроизводимость и масштабируемость решения.
